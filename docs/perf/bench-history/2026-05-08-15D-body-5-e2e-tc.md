# Stage 15.D-body.5 — E2E bench, tensor-core hybrid: 0% lift (negative)

Date: 2026-05-08 (commit b9bc05f)
Device: NVIDIA GeForce RTX 4060 Laptop GPU (sm_89)
Build: `cargo build --release -p vllm-server --features cuda-full`
Server: `VLLM_AWQ_HYBRID=1 vllm-server serve --model Qwen/Qwen3-4B-AWQ --port 8765 --max-model-len 1024 --num-blocks 256`
Bench: `python3 scripts/bench_decode.py --runs 3 --concurrency 1 4 8 --prompt-len 256 --max-tokens 256`

## Result

| Concurrency | Hybrid (TC) tps | Baseline tps (memory) | Lift |
|---|---|---|---|
| c=1  | 42.4 | 43 | 0% |
| c=4  | 63.7 | 65 | 0% |
| c=8  | crash (KV exhaustion @ 256 blocks × 8 reqs × 512 tok) | 64 | n/a |

c=1 is expected to tie (M=1 below `HYBRID_M_MIN=4`, falls through to
production marlin_gemm). c=4 should have shown the projected ~2× lift
based on body.4 microbench data (TC = 3.19× over prod at M=4 on
`mlp_up_old`, 2.13× on `down`). It didn't.

## This replays Stage 15.E.3 hypothesis 2

15.E.3 documented the same negative result for the **software** tile-mma
path: microbench claimed 1.10–1.24× per-call wins, e2e showed 0% lift.
The unresolved hypothesis was:

> per-call launch overhead amortisation differs in microbench vs real
> workload — would need nvprof under production load to settle.

Body.5 confirms the same shape with the **tensor-core** path: 3–5×
microbench wins, **still 0% e2e lift**. Whatever overhead is masking the
tile_mma improvement is shape- and magnitude-independent; it's
structural to the tile_mma dispatch.

## Likely culprits (untested)

1. **CustomOp1 framework overhead.** Each `dispatch_marlin_tile_mma_v1`
   call wraps a `CustomOp1` impl, walks candle's apply_op1 path,
   constructs a tensor for the result. The production marlin_gemm
   bypasses this via the same code path — but the relative weight of
   per-call book-keeping vs kernel time may be larger for tile_mma
   because the tile_mma kernel itself is fast.
2. **Cold-cache effects.** The microbench warms up; production decode
   keeps switching between attention, RMSNorm, RoPE, MoE, and matmul
   kernels — code-cache and register-file contention not modelled in
   the isolated microbench.
3. **Stream synchronisation.** tile_mma uses the default stream;
   adjacent kernels (attention, RMSNorm) may also use the default
   stream and serialise differently than the microbench loop.

## Next steps (not in this commit)

- **nvprof / Nsight Compute** the production path to identify which
  kernel(s) are queue-blocked or spend time in launch overhead.
- **Compare kernel duration** in production vs microbench timeline —
  if same, overhead is upstream; if different, environmental.
- If overhead is structural, consider a fused multi-layer dispatcher
  that batches kernel launches — but this is a separate Stage.

## What is preserved by this work

The full tile-mma producer/dispatcher/storage/forward-gate stack now
runs through tensor cores with **2-5× kernel-level speed** over
production marlin_gemm. When the dispatch overhead is fixed (or the
underlying scheduler reduces per-layer Python/Rust hop cost), the gain
will materialise.

Decision: keep `VLLM_AWQ_HYBRID=1` opt-in (no e2e win at present, no
regression), document this finding, **stop iterating on this gate**
until per-call overhead is investigated. Stage 15.D-body achieved its
correctness goals (production-shape contract on tensor cores, 12/12 GPU
tests pass at TC) but not its e2e perf goal (0% vs +38%/+72% projected).
