# Substep 2.2 — Async DtoH (validated, marginal lift on tensor-core baseline)

Date: 2026-05-09 (post-resolution)
Device: NVIDIA GeForce RTX 4060 Laptop GPU (sm_89), WSL2
Server: `VLLM_AWQ_HYBRID=1 [VLLM_ASYNC_SAMPLING=1] ./target/release/vllm-server serve --model Qwen/Qwen3-4B-AWQ --port 8765 --enforce-eager --max-model-len 1024 --num-blocks 384`
Bench: `python3 scripts/bench_decode.py --concurrency 1 4 8 --prompt-len 256 --max-tokens 128 --runs 3`

## Verdict: code correct, +0.8% lift on tensor-core AWQ baseline

The "WSL2 stream serialisation" finding from the earlier snapshot
(`2026-05-09-substep-2.2-async-dtoh-wsl2-neutral.md`) was incorrect
— that bench was run **without** `VLLM_AWQ_HYBRID=1`, which puts
AWQ through a non-tensor-core slow path and inflates per-step
forward time so much that sampler stall is dwarfed.

With the baseline running tensor-core AWQ correctly:

| c | HYBRID only (baseline) | HYBRID + VLLM_ASYNC_SAMPLING=1 |
|---|---|---|
| 1 | 45.9 | 46.7 |
| 4 | 85.8 | 86.3 |
| 8 | 118.8 | **119.8 (+0.8%)** |

Async DtoH delivers **+0.8% c=8** — within bench noise. The reason:
on the tensor-core path, sampler is already only ~6 ms / 115 ms step
(decode_profile slot, c=8). Even fully eliminating that stall would
yield ≤6% lift, and async DtoH only amortises across multi-step
blocks, not eliminates.

## Forward dominates the new baseline (decode_profile breakdown, c=8)

Per-layer (μs/call, 36 layers):
- in_norm: 5
- qkv: 369
- qk_norm: 7.5
- rope: 43
- cache_w: 11
- paged_attn: 175
- o_proj: 236
- post_norm: 5
- **mlp: 1620 (65 % of layer)**
- layer_sum: 2472

Per-forward (μs):
- embed: ~50
- final_norm: 5
- **lm_head_evt: 26,886 (≈ 27 ms)**

So forward total ≈ 89 ms (layers) + 27 ms (lm_head) ≈ 116 ms.
Sampler 6 ms, alloc + metadata < 1 ms.

**Real bottleneck shifted to forward MLP (AWQ-Marlin gate/up/down)
and lm_head matmul.**

## What this means for Substep 2.2

- Code is correct, env-gated, passes tests.
- Lift on this hardware/env is marginal (+0.8% c=8) — the
  `to_vec1()` stall measured at 50 ms in earlier `nsys` snapshot
  no longer applies (it was dominated by forward queue depth on
  the slow non-TC path; with TC it's just kernel time).
- Land it as infrastructure but DO NOT recommend `VLLM_ASYNC_SAMPLING=1`
  by default — overhead of two streams is real, lift is not.
- Future: if forward/lm_head get optimised significantly, sampler
  share grows and async DtoH may become worth defaulting on.
