## What's now possible that wasn't

Pre-fixes the c=16 case **hard-deadlocked the KV cache**:
`num_running=0 num_waiting=N gpu_cache_usage_perc=100.0` with no
progress until SIGKILL. Two bugs gated this:

1. **KV block leak in `finish_request_with_error_deferred`** (commit
   `1f9a198`). When prefill / decode failed, the engine removed the
   ActiveRequest from the map without freeing its block_table.
   Cascading prefill OOMs at c=16 leaked all 384 blocks in seconds.
2. **UTF-8 panic at `helpers.rs:1527`** (commit `40c595e`). Streaming
   non-ASCII output (Qwen3 Chinese responses) panicked when
   `streamed_text_len` landed mid-multibyte-char on non-monotonic
   tokenizer decode. This crashed the engine task during the c=8 run
   (compounding the c=16 problem).

After both fixes c=16 runs cleanly, hits 185 tps aggregate, and the
server stays healthy across the multi-concurrency sweep.

## Aggregate scaling vs baseline

Baseline at c=8 (pre-fix, software-only, no hybrid) was ~64 tps. With
hybrid+TC enabled and the leak/panic gone, c=16 reaches **185 tps —
2.89× the pre-stage baseline c=8 throughput**. Stage 15.D-body's
production goal (close the gap to Python vLLM's 351 tps at c=8) is
now within ~1.9× — substantial closure of the original 5.5× gap.

## Decisions

1. Both fixes ship as PRODUCTION fixes (not behind any gate). The
   leak path was hit by ANY failed prefill/decode, not just hybrid.
2. `VLLM_AWQ_HYBRID=1` opt-in stays — VRAM cost still real.
3. c=16 throughput unlocks meaningful concurrent production deploys
   on 8 GiB cards.
