## What 2.1 changes

- `execute_batched_decode_with_graph` now takes `prev_sampled_ids: Option<&Tensor>`
  and returns `(Vec<(RequestId, String)>, Option<Tensor>)`.
- On no-adapter, no-preemption path, when prev tensor's leading dim
  matches `batch_size`, it is reshaped to `[B,1]` and used directly as
  `input_ids` — bypassing the host `Tensor::from_vec(token_ids, ...)`
  round-trip.
- GPU sampler call switched from `gpu_sample_batch_with_diffs` (Vec<u32>
  return) to `gpu_sample_batch_with_diffs_to_tensor` (Tensor return).
  The caller still does `to_vec1::<u32>()` for `generated_token_ids`
  bookkeeping — that's the 2.2 work.
- `standard.rs::execute_decodes` threads `Option<Tensor>` through the
  multi-step loop; drops it on any active-set size shrink (preemption
  / EOS / max_tokens).

This is plumbing only. The actual lift comes when 2.2 moves the
`to_vec1` DtoH onto a separate stream so the next forward overlaps.

## Note on absolute throughput

The c=8 baseline number (67.8 tps) is **lower** than the post-15.D-body
snapshot (`2026-05-08-15D-body-5-e2e-tc-fixed.md` = 111.3 tps, same
parameters). Possible causes (not investigated as part of 2.1):

1. GPU clock-down under sustained load on this laptop GPU
   (memory rule `feedback_bench_runs.md` calls this out generally).
2. Environmental drift since that snapshot's commit (other commits
   landed on main; one might have regressed unrelated work).
3. Background load (browser, IDE) competing for shared resources.

The 2.1 vs baseline comparison is on the **same machine in the same
session** using stash/pop to swap between the two source states, so
the comparison is apples-to-apples regardless of absolute number.
Sleeping the absolute regression hunt for now — the 2.2 lift will be
measured against this baseline.
