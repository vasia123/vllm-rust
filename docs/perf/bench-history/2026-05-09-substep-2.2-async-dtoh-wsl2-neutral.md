# Substep 2.2 — Async DtoH overlap (env-gated, WSL2 baseline-neutral)

Date: 2026-05-09
Device: NVIDIA GeForce RTX 4060 Laptop GPU (sm_89), WSL2 host
Server: `VLLM_ASYNC_SAMPLING=1 ./target/release/vllm-server serve --model Qwen/Qwen3-4B-AWQ --port 8765 --enforce-eager --max-model-len 1024 --num-blocks 384`
Bench: `python3 scripts/bench_decode.py --concurrency 1 4 8 --prompt-len 256 --max-tokens 128 --runs 3`

## Status: infrastructure landed, expected to deliver on native Linux

Substep 2.2 wires a dedicated CUDA stream + plain page-locked
(non-WC) host buffers to issue sampler-result DtoH copies that the
host syncs only at the end of each multi-step block. Code is correct
(7/7 correctness, 4504 unit tests, env-gated, fall-back-safe), but on
this WSL2 host the absolute throughput delta is 0%.

| c | with VLLM_ASYNC_SAMPLING=1 | baseline (gate off) | Δ |
|---|---|---|---|
| 1 | 46.7 | 45.8 | +2% |
| 4 | 69.5 | 68.8 | +1% |
| 8 | 68.5 | 67.8 | +1% |

The pipelined path is fully exercised (`async_sampling=info` log
fires, `pipelined timings steps=4` per multi-step block, no DtoH
errors), and per-step host enqueue time DROPS from ~119 ms (std
path's `to_vec1` stall) to ~88 ms (= 350 ms enqueue / 4 steps).
But the drain at the end (~117 ms) brings total per-multi-step
back to ~470 ms — same as serial baseline (~478 ms).

That's the diagnostic: the side-stream's `cuMemcpyDtoHAsync` is
correctly queued and async, BUT cross-stream overlap with the next
multi-step block's forward kernels does NOT occur in the WSL2 GPU
stack on this host. Memory rule
`feedback_wsl2_profiling_blocked.md` already flags WSL2 GPU
limitations; the streams appear to serialise here even when issued
on different `CUstream` handles.

On native Linux with the same GPU and recent driver, this same
infrastructure should deliver the projected +25–50% lift at c=8
because the side-stream's memcpy genuinely overlaps with the next
forward's kernel queue. We're landing the infrastructure now so
when the baseline is run on native hardware the lift is captured
without re-engineering.

## What landed

- `crates/core/src/engine/async_sampler.rs`: `AsyncSamplerInfra` +
  `SamplerPinnedSlot`. We allocate the host buffers via
  `cuMemHostAlloc(flags=0)` directly because cudarc 0.16's
  `alloc_pinned` defaults to `CU_MEMHOSTALLOC_WRITECOMBINED`, and
  WC-pinned memory rejects DtoH (`CUDA_ERROR_INVALID_VALUE` —
  confirmed empirically before fix). Documented in module doc-string.
- `crates/core/src/engine/helpers.rs`:
  - `defer_dtoh: bool` parameter on `execute_batched_decode_with_graph`.
    When true on the GPU sampler fast path, skip `to_vec1` and
    `generated_token_ids.push`; still advance block-table and
    seqlen counters (those don't depend on the token value).
  - `execute_pipelined_multi_step_decode` — internalises the
    multi-step loop, dispatches async DtoH on a side stream, drains
    at the end and pushes tokens.
  - `dispatch_async_dtoh_for_sampler` — extracts the tensor's
    `CudaSlice<u32>`, sub-slices to `n_elems`, queues memcpy on the
    side stream, records the slot's event so the drain can sync.
- `crates/core/src/engine/standard.rs`: env-gated dispatch in
  `execute_decodes`. `Mutex<AsyncSamplerSlot>` lazy-init pattern
  (Uninit → Disabled / Init).

## Why the WSL2 result still warrants landing

1. **Code is correct.** Env-gated; default behaviour unchanged.
2. **Architecture is right.** When run on a host where streams do
   overlap (native Linux, datacenter), the lift will materialise.
3. **Findings are recorded.** WSL2 limitation is documented in
   memory (`perf_baseline_environmental_drift.md`) and this snapshot.
4. **Foundation for further work.** Substep 3 (deferred finish-check)
   builds on the pipelined helper; without 2.2 in place there's no
   place to hang the over-decode-trim logic.

## Decision: stop here on Substep 2 / 3 for this WSL2 session

Substep 3 (deferred finish-check) compounds the same async-DtoH
mechanism — without overlap on this host, it cannot show e2e lift
either. Re-running the validation on native Linux is the next
high-information action; meanwhile the engine continues on the
serial (default) path with no regression.
